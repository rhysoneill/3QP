# Module 09: Language & Log Output System — Technical Specification

## 1. Overview

This specification defines the complete logging architecture for the 3QP Project. It establishes the rules, schemas, formats, and lifecycle management protocols for all structured log output produced by simulation modules.

## 2. Log Categories

### 2.1 State Logs

**Purpose**: Capture point-in-time snapshots of module internal state variables.

**Production Trigger**: Generated at configurable intervals aligned with TQP Core time-steps.

**Required Fields**:
- `log_type`: Fixed value "STATE"
- `timestamp`: Simulation time-step index (integer)
- `wall_time`: System clock timestamp (ISO 8601 format)
- `module_id`: Identifier of producing module (string)
- `state_snapshot`: Dictionary of state variable name-value pairs
- `log_version`: Schema version identifier (semantic version string)

**Optional Fields**:
- `snapshot_id`: Unique identifier for this snapshot
- `delta_encoding`: Boolean flag indicating incremental state encoding
- `compression_method`: String describing compression applied to state data

**Granularity**: Configurable per module, default one snapshot per simulation time-step.

### 2.2 Event Logs

**Purpose**: Record discrete occurrences, transitions, and threshold crossings.

**Production Trigger**: Generated immediately upon event detection by module.

**Required Fields**:
- `log_type`: Fixed value "EVENT"
- `timestamp`: Simulation time-step index (integer)
- `wall_time`: System clock timestamp (ISO 8601 format)
- `module_id`: Identifier of producing module (string)
- `event_category`: Enumerated event type identifier (string)
- `event_data`: Dictionary of event-specific attributes
- `log_version`: Schema version identifier (semantic version string)

**Optional Fields**:
- `event_id`: Unique identifier for this event occurrence
- `causality_chain`: Array of prior event IDs contributing to this event
- `severity_level`: Enumerated priority indicator
- `related_entity_ids`: Array of agent or entity identifiers involved

**Granularity**: One log entry per discrete event occurrence.

### 2.3 Cycle Logs

**Purpose**: Document TQP Core execution metadata for each simulation cycle.

**Production Trigger**: Generated by TQP Core at cycle completion.

**Required Fields**:
- `log_type`: Fixed value "CYCLE"
- `timestamp`: Simulation time-step index (integer)
- `wall_time`: System clock timestamp (ISO 8601 format)
- `cycle_duration_ms`: Wall-clock duration of cycle execution (float)
- `modules_executed`: Array of module identifiers executed this cycle
- `cycle_status`: Enumerated completion status ("COMPLETE", "PARTIAL", "ERROR")
- `log_version`: Schema version identifier (semantic version string)

**Optional Fields**:
- `module_timings`: Dictionary mapping module IDs to execution durations
- `memory_usage_mb`: Peak memory consumption during cycle
- `error_records`: Array of error objects if cycle_status indicates failure

**Granularity**: One log entry per TQP Core time-step cycle.

### 2.4 Metric Logs

**Purpose**: Aggregate numerical measurements and statistical summaries.

**Production Trigger**: Generated at configurable intervals or window boundaries.

**Required Fields**:
- `log_type`: Fixed value "METRIC"
- `timestamp_start`: Simulation time-step index for window start (integer)
- `timestamp_end`: Simulation time-step index for window end (integer)
- `wall_time`: System clock timestamp (ISO 8601 format)
- `module_id`: Identifier of producing module (string)
- `metric_name`: Identifier for measured quantity (string)
- `metric_value`: Numerical measurement or aggregate (float or integer)
- `log_version`: Schema version identifier (semantic version string)

**Optional Fields**:
- `metric_unit`: Physical or logical unit of measurement (string)
- `aggregation_method`: Method used for computing metric ("SUM", "MEAN", "MAX", etc.)
- `sample_count`: Number of observations aggregated
- `confidence_bounds`: Dictionary with statistical confidence interval data

**Granularity**: Configurable aggregation windows, default 10 time-steps.

## 3. Format Specifications

### 3.1 Field Naming Conventions

- All field names use lowercase with underscore separators (snake_case)
- No abbreviations except standard units (ms, mb, etc.)
- Boolean fields prefixed with "is_" or "has_" where appropriate
- Timestamp fields always include temporal scope in name

### 3.2 Serialization Format

**Primary Format**: JSON Lines (newline-delimited JSON)

Each log entry is serialized as a single-line JSON object followed by newline character. This enables:
- Streaming write operations without array closing
- Sequential read with simple line parsing
- Append-only file operations

**Example Structure**:
```
{"log_type":"STATE","timestamp":42,"module_id":"mod_01","state_snapshot":{...},"log_version":"1.0.0"}
{"log_type":"EVENT","timestamp":43,"module_id":"mod_02","event_category":"threshold_cross","event_data":{...},"log_version":"1.0.0"}
```

**Alternative Formats**: Specification allows pluggable serialization backends:
- Binary MessagePack for size-constrained environments
- Columnar Parquet for analytical query workloads
- CBOR for compact representation with schema validation

### 3.3 Data Type Constraints

- **Integers**: 64-bit signed, representing time-steps or counts
- **Floats**: IEEE 754 double precision for all decimal values
- **Strings**: UTF-8 encoded, maximum length 4096 characters per field
- **Dictionaries**: Nested depth limited to 5 levels
- **Arrays**: Maximum 10,000 elements per array field
- **Booleans**: Explicit true/false, no implicit truthy values
- **Null**: Permitted only for optional fields, never for required fields

### 3.4 Timestamp Synchronization

All `timestamp` fields reference the TQP Core simulation clock:
- Integer values starting at 0 for simulation initialization
- Monotonically increasing with each time-step
- Increment by 1 per cycle (variable wall-clock duration)

All `wall_time` fields use ISO 8601 format with millisecond precision:
- Format: `YYYY-MM-DDTHH:MM:SS.sssZ`
- Timezone: UTC only
- Purpose: External timeline anchoring and performance measurement

## 4. Log Granularity and Timing Rules

### 4.1 State Log Timing

- State snapshots generated at end of module update execution
- Snapshot captures post-update state before next cycle begins
- Configurable decimation: record every Nth time-step where N ≥ 1
- Default: Full state snapshot every time-step for modules with <100 state variables

### 4.2 Event Log Timing

- Event logs generated synchronously with event detection
- No buffering or delayed emission
- Multiple events in single time-step produce multiple log entries
- Ordering: events logged in detection sequence within time-step

### 4.3 Cycle Log Timing

- Cycle log generated after all module updates complete
- Timestamp reflects the completed cycle number
- Generated even for empty cycles (no module updates)

### 4.4 Metric Log Timing

- Generated at window boundary closure
- Window boundaries aligned to time-step boundaries
- Non-overlapping windows preferred for statistical independence
- Configurable window size per metric type

## 5. Data Schemas

### 5.1 Schema Versioning

Each log entry includes `log_version` field specifying schema version:
- Format: Semantic versioning (MAJOR.MINOR.PATCH)
- Backward compatibility guaranteed within MAJOR version
- Schema changes increment version appropriately
- Version registry maintained separately from log files

### 5.2 Required Field Validation

At log ingestion, system validates:
- All required fields present
- Field data types match specification
- Timestamp values are non-negative integers
- Module IDs reference registered modules
- Enumerated values match allowed sets

**Validation Failure Handling**:
- Invalid log entry rejected
- Error logged to system error stream
- Originating module notified of validation failure
- Simulation continues unless critical error threshold exceeded

### 5.3 Optional Field Handling

- Optional fields may be omitted entirely (not present in JSON)
- If present, optional fields MUST conform to type specifications
- Optional field with unknown value represented by omission, not null
- New optional fields may be added without version increment

### 5.4 Extension Fields

Modules may include custom extension fields:
- Field names must use prefix `ext_` followed by module identifier
- Example: `ext_mod02_custom_metric`
- Extension fields are optional and not validated by core logging system
- Documentation of extension fields is module's responsibility

## 6. Batch Processing and Compression

### 6.1 Batch Boundaries

Logs batched into discrete files based on:
- Time-step ranges (e.g., steps 0-999, 1000-1999)
- Module grouping (all logs from specific module)
- Log type (separate files for STATE, EVENT, etc.)

Default batching strategy: 1000 time-steps per file, grouped by log type.

### 6.2 File Naming Convention

Pattern: `{module_id}_{log_type}_{timestamp_start}_{timestamp_end}.jsonl`

Examples:
- `mod_01_STATE_00000000_00000999.jsonl`
- `mod_02_EVENT_00001000_00001999.jsonl`
- `core_CYCLE_00000000_00000999.jsonl`

### 6.3 Compression

- Optional gzip compression applied to completed batch files
- Compressed files use `.jsonl.gz` extension
- Compression applied only after batch file closed (no appends possible)
- Default: compression enabled for batches exceeding 1 MB uncompressed

### 6.4 Streaming vs Batch Mode

**Streaming Mode**: Logs written to active batch file immediately upon generation
- Lowest latency
- Higher I/O overhead
- Suitable for real-time monitoring

**Batch Mode**: Logs buffered in memory and flushed at intervals
- Configurable flush interval (default: 100 entries or 1 second)
- Reduced I/O overhead
- Risk of data loss on crash

Default: Streaming mode for EVENT logs, batch mode for STATE and METRIC logs.

## 7. Log Lifecycle

### 7.1 Creation Phase

1. Module generates log data structure
2. Module calls logging system registration interface
3. Logging system validates schema compliance
4. Valid log serialized to current batch file
5. Confirmation or error returned to module

### 7.2 Active Phase

- Batch file remains open for append operations
- File system lock prevents external modification
- In-progress flag maintained in index manifest
- Periodic flush to disk based on buffering mode

### 7.3 Rotation Phase

Triggered when batch boundary reached:
1. Current batch file closed
2. Optional compression applied
3. Index manifest updated with batch metadata
4. New batch file created for subsequent logs
5. Old batch file marked read-only

### 7.4 Retention Phase

- Completed batch files stored in retention directory
- Retention period configurable per log type (default: unlimited)
- Automated deletion after retention period expires
- Option to archive to external storage before deletion

### 7.5 Export Phase

On-demand or scheduled export:
1. Query specifies time-step range and/or module filter
2. System locates relevant batch files
3. Logs extracted and optionally filtered
4. Export package created (directory or compressed archive)
5. Export manifest generated with provenance metadata

## 8. Integration with TQP Core

### 8.1 Time-Step Synchronization

TQP Core provides synchronization signals:
- `cycle_start(timestamp)`: Signals beginning of new time-step
- `cycle_end(timestamp)`: Signals completion of time-step

Logging system aligns operations:
- Event logs may occur during cycle
- State logs finalized at cycle end
- Cycle log generated after all modules complete

### 8.2 Module Registration

Modules register with logging system during initialization:
- Provide module ID and version
- Declare log types produced (STATE, EVENT, METRIC)
- Specify schema extensions if used
- Receive logging interface handle

### 8.3 Logging Interface

Standard interface methods:
- `log_state(timestamp, state_dict)`: Submit state snapshot
- `log_event(timestamp, event_category, event_data)`: Submit event
- `log_metric(timestamp_start, timestamp_end, metric_name, metric_value)`: Submit metric

Interface enforces required fields and type validation.

## 9. Error Handling

### 9.1 Malformed Log Entries

On schema validation failure:
- Entry rejected and not written to batch file
- Error record generated to error log stream
- Error includes module ID, timestamp, and validation failure reason
- Simulation continues unless error rate exceeds threshold (default: 100 errors per 1000 time-steps)

### 9.2 Missing Logs

Detection mechanisms:
- Expected log count comparison based on registered modules
- Time-step gap detection in batch files
- Module heartbeat monitoring

Response:
- Gap recorded in index manifest
- Warning logged to error stream
- Downstream analysis tools notified of incomplete data

### 9.3 I/O Failures

On disk write failure:
- Attempt retry with exponential backoff (up to 3 attempts)
- If retry fails, buffer logs in memory
- If memory buffer exceeds limit (default: 10,000 entries), begin dropping oldest logs
- Critical error flag set, TQP Core notified

### 9.4 Corruption Detection

Each batch file includes:
- Header with schema version and creation timestamp
- Footer with entry count and checksum
- Verification performed on batch rotation

On corruption detection:
- Corrupt file quarantined
- Last known good batch identified
- Simulation may be rolled back depending on severity

## 10. Extensibility

### 10.1 New Log Categories

Adding new log category requires:
- Define required and optional fields
- Specify schema version
- Update log type enumeration
- Register with logging system core
- No modification to existing log files required

### 10.2 Schema Evolution

Field additions:
- New optional fields: no version change required
- New required fields: increment MINOR version
- Field type changes: increment MAJOR version

Backward compatibility:
- Readers must tolerate unknown optional fields
- Readers must provide defaults for new required fields in old logs
- Version-specific parsers maintained for each MAJOR version

### 10.3 Custom Serialization Backends

Pluggable serialization interface:
- Implement `serialize(log_dict)` and `deserialize(byte_stream)` methods
- Register backend with format identifier
- Configure per log type or globally
- All backends must preserve field semantics exactly

## 11. Requirements for Downstream Consumers

### 11.1 Readability

- Logs are self-describing through field names
- No external schema file required for basic parsing
- Human inspection possible using standard JSON tools
- Log version field enables version-aware parsing

### 11.2 Consistency

- Timestamp fields always present and correctly ordered
- No duplicate entries for same (module_id, timestamp, log_type) tuple
- Field semantics consistent across simulation run
- Batch file boundaries do not affect logical consistency

### 11.3 Reproducibility

- Logs contain sufficient information to reconstruct module state trajectories
- Deterministic ordering within time-steps
- No randomness in log generation or serialization
- Identical simulation configuration produces identical logs (excluding wall_time)

### 11.4 Queryability

- Logs support standard filtering predicates on required fields
- Batch file organization enables efficient time-range queries
- Index manifests provide metadata for query optimization
- No specialized query language required for basic access

## 12. Performance Considerations

### 12.1 Write Throughput

Expected load:
- 100-1000 log entries per time-step
- 10-100 time-steps per second
- Target: sustain 100,000 log writes per second

Optimization strategies:
- Batch mode with memory buffering
- Asynchronous I/O where supported
- Pre-allocated file space to reduce fragmentation

### 12.2 Storage Efficiency

Size estimates:
- State log: 100-1000 bytes per entry (compressed: 20-200 bytes)
- Event log: 50-500 bytes per entry (compressed: 10-100 bytes)
- Cycle log: 200-500 bytes per entry (compressed: 40-100 bytes)
- Metric log: 50-200 bytes per entry (compressed: 10-40 bytes)

1 million time-steps, 100 entries/step: ~10 GB uncompressed, ~2 GB compressed

### 12.3 Query Performance

Index manifest enables:
- O(log n) time-step range lookup
- O(1) module-specific batch identification
- Parallel loading of multiple batch files

Target: Load and filter 1 million log entries in <10 seconds on standard hardware

## 13. Validation and Testing

Validation of logging system implementation requires:

- Schema compliance testing for all log categories
- Timestamp ordering verification
- Batch boundary correctness checks
- Compression/decompression round-trip validation
- I/O failure recovery simulation
- Performance benchmarking against targets in Section 12

Testing must be performed independently without generation of scenario-specific content or narrative examples.
